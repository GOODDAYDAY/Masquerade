# REQ-011 Werewolf Prompt Intelligence Upgrade

> Status: Completed
> Created: 2026-03-16
> Updated: 2026-03-16

## 1. Background

首次完整狼人杀游戏回放暴露了 AI 策略质量严重不足：

- **狼人目标太可预测：** 每轮都选"最显眼"的人，守卫用同样逻辑保护，导致连续 3 轮完美守中
- **狼人讨论过于简单：** 2 轮讨论全是重复的"指向→点头→确认"，缺乏战术深度
- **女巫过于保守：** 5 轮游戏毒药从未使用，错失关键机会
- **预言家不会带队：** 查到狼人却无法让好人相信，被投票淘汰
- **守卫缺乏博弈思维：** 纯猜测保护，没有分析狼人心理
- **村民投票没逻辑：** 3 轮投票全投好人，0 个狼人被投出
- **猎人死亡时未开枪：** 猎人被击杀后没有触发开枪机制（需确认是 prompt 还是 engine 问题）

核心原因：各角色的 thinker prompt 策略引导太简单，缺乏游戏深度。

## 2. Target Users & Scenarios

- **开发者/用户：** 希望 AI 狼人杀对局质量更高、更有观赏性

## 3. Functional Requirements

### F-01 狼人 Night Prompt 升级

- **选目标策略丰富化：**
  - 分析守卫可能保护谁（上轮保护了谁、这轮不能连续保护、守卫倾向保护谁）
  - 分析女巫是否还有解药（第一晚用过了吗？）
  - 避开"显然"目标，引入反向思维（"大家都觉得我们会刀预言家，不如刀个不起眼的"）
  - 考虑白天投票后的存活阵营比例
- **讨论内容升级：**
  - 第一轮讨论：提出 2-3 个候选目标及理由
  - 第二轮讨论：回应队友动作，形成共识或分歧处理
  - 不要每轮都是"指向→点头"的重复模式

### F-02 狼人 Day Prompt 升级

- 伪装时考虑"跳身份"策略（假装预言家、假装守卫等）
- 如何应对被查验的情况
- 投票时如何分票、如何引导好人互投
- 分析场上已暴露的信息，利用信息差

### F-03 预言家 Prompt 升级

- **Night：** 首验策略更有深度（验可能的狼人而非随机）
- **Day：** 何时跳身份？跳了之后如何自证？如何传递查验结果让人信服？
- 查到狼人后如何带队投票
- 查到好人后如何利用金水链

### F-04 女巫 Prompt 升级

- **解药使用策略：** 什么时候该救（关键角色被刀）、什么时候不救（疑似自刀骗药）
- **毒药使用策略：** 不要一直 skip！有合理嫌疑就应该考虑毒。分析白天发言锁定目标
- 毒药和解药同一晚不能同时使用的策略考量
- 什么时候该暴露女巫身份

### F-05 守卫 Prompt 升级

- **反向博弈：** 分析狼人心理——"狼人知道我可能保护谁，所以他们会避开"→"所以我反而应该保护那个人"
- 分析上轮是否守中（如果守中了说明判断对，下轮换人；没守中说明判断错，调整策略）
- 不要每轮都保护"最显眼"的人
- 考虑自守的时机

### F-06 村民 Prompt 升级

- 分析发言逻辑链——谁的发言自洽、谁前后矛盾
- 如何判断预言家真假（对跳时如何选择站边）
- 投票时形成归票共识，不分散票数
- 关注沉默玩家和跟风投票的玩家

### F-07 猎人 Prompt 升级

- 死亡时开枪决策更果断（有合理怀疑就开枪，别浪费技能）
- 白天如何隐藏身份但又能暗示自己有身份
- 被投票时是否应该亮身份保命

### F-08 夜晚死亡公告事件

- **问题：** 当游戏在夜晚结算后直接结束（狼 ≥ 好人），该轮 0 speaks、0 votes，前端从夜晚行动直接跳到结局，缺少"昨晚 xxx 被杀"的过渡
- **修复：** 每轮夜晚结算后，如果有死亡，在 round.events 中插入一个"死亡公告"事件（action.type = "death_announce"），包含死亡玩家列表
- **Runner 改动：** 夜晚结算后、白天讨论前，从 engine.get_public_state() 读取 night_deaths，生成 death_announce 事件
- **前端改动：** ActionScene 已有通用兜底，death_announce 自动展示。可在 ACTION_LABELS 中添加映射
- **边界：** 即使游戏在此时结束（不进入白天），也要记录这个事件

### F-09 守卫连续保护的信息透传

- **问题：** 守卫的 private_info 中有 `last_protected`，但在白天讨论时，守卫无法判断"昨晚是否守中了"（因为不知道谁死了、没人死是因为自己守中还是女巫救了）
- **现状：** 这其实是正确的——守卫在真实游戏中也不知道是否守中。无需修改
- **但 prompt 可以引导：** 守卫可以根据"今天是平安夜还是有人死亡"来间接推断自己是否守中

## 4. Non-functional Requirements

- **NF-01：** 不新增 LLM 调用次数（prompt 优化不增加调用）
- **NF-02：** F-08 的 Runner 改动需向后兼容（spy 游戏不受影响）
- **NF-03：** 现有测试全部通过

## 5. Out of Scope

- 谁是卧底的 prompt 优化
- Agent 框架修改
- Engine 规则修改
